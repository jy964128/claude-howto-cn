# 检查点与回退

检查点允许你保存对话状态并回退到 Claude Code 会话中的先前位置。这对于探索不同方法、从错误中恢复或比较替代方案非常有用。

## 概述

检查点允许你保存对话状态并回退到之前的某个位置，从而安全地进行实验和探索多种方法。它们是对话状态的快照，包括：
- 所有已交换的消息
- 已进行的文件修改
- 工具使用历史
- 会话上下文

在探索不同方法、从错误中恢复或比较替代方案时，检查点非常宝贵。

## 核心概念

| 概念 | 描述 |
|---------|-------------|
| **检查点** | 对话状态的快照，包括消息、文件和上下文 |
| **回退** | 返回到之前的检查点，丢弃后续的更改 |
| **分支点** | 从中探索多种方法的检查点 |

## 访问检查点

你可以通过两种主要方式访问和管理检查点：

### 使用键盘快捷键
按两次 `Esc`（`Esc` + `Esc`）打开检查点界面并浏览已保存的检查点。

### 使用斜杠命令
使用 `/rewind` 命令（别名：`/checkpoint`）快速访问：

```bash
# 打开回退界面
/rewind

# 或使用别名
/checkpoint
```

## 回退选项

当你执行回退时，会看到一个包含五个选项的菜单：

1. **恢复代码和对话** -- 将文件和消息都恢复到该检查点的状态
2. **恢复对话** -- 仅回退消息，保持当前代码不变
3. **恢复代码** -- 仅回退文件更改，保留完整的对话历史
4. **从此处摘要** -- 从此时间点开始将对话压缩为 AI 生成的摘要，释放上下文窗口空间。所选时间点之前的消息保持不变。磁盘上的文件不会更改。原始消息保留在会话记录中。你可以选择提供指示，使摘要聚焦于特定主题。
5. **取消** -- 取消并返回当前状态

> **注意**：恢复对话或进行摘要后，所选消息的原始提示会恢复到输入框中，以便你可以重新发送或编辑它。

## 自动检查点

Claude Code 会自动为你创建检查点：

- **每次用户输入** - 每次用户输入时都会创建一个新的检查点
- **持久化** - 检查点在会话之间持久保存
- **自动清理** - 检查点在 30 天后自动清理

这意味着你随时可以回退到对话中的任何一个先前位置，无论是几分钟前还是几天前。

## 使用场景

| 场景 | 工作流程 |
|----------|----------|
| **探索不同方法** | 保存 → 尝试方法 A → 保存 → 回退 → 尝试方法 B → 比较 |
| **安全重构** | 保存 → 重构 → 测试 → 如果失败：回退 |
| **A/B 测试** | 保存 → 设计方案 A → 保存 → 回退 → 设计方案 B → 比较 |
| **错误恢复** | 发现问题 → 回退到上一个良好状态 |

## 使用检查点

### 查看和回退

按两次 `Esc` 或使用 `/rewind` 打开检查点浏览器。你会看到一个带有时间戳的所有可用检查点列表。选择任意检查点即可回退到该状态。

### 检查点详情

每个检查点显示：
- 创建时的时间戳
- 被修改的文件
- 对话中的消息数量
- 使用过的工具

## 实际示例

### 示例 1：探索不同方法

```
用户：让我们为 API 添加一个缓存层

Claude：我将为你的 API 端点添加 Redis 缓存……
[在检查点 A 进行了更改]

用户：等等，不如试试内存缓存

Claude：我将回退以探索另一种方法……
[用户按 Esc+Esc 并回退到检查点 A]
[在检查点 B 实现内存缓存]

用户：现在我可以比较两种方法了
```

### 示例 2：从错误中恢复

```
用户：将认证模块重构为使用 JWT

Claude：我将重构认证模块……
[进行了大量更改]

用户：等等，这破坏了 OAuth 集成。我们回退吧。

Claude：我将帮你回退到重构之前……
[用户按 Esc+Esc 并选择重构前的检查点]

用户：这次我们试试更保守的方法
```

### 示例 3：安全实验

```
用户：让我们试试用函数式风格重写这个
[在实验前创建检查点]

Claude：[进行了实验性更改]

用户：测试失败了。我们回退吧。
[用户按 Esc+Esc 并回退到检查点]

Claude：我已经回退了更改。让我们试试另一种方法。
```

### 示例 4：分支方法

```
用户：我想比较两种数据库设计
[记下检查点——称之为 "开始"]

Claude：我将创建第一种设计……
[实现了模式 A]

用户：现在让我返回去尝试第二种方法
[用户按 Esc+Esc 并回退到 "开始"]

Claude：现在我将实现模式 B……
[实现了模式 B]

用户：太好了！现在我有两种模式可供选择
```

## 检查点保留策略

Claude Code 会自动管理你的检查点：

- 每次用户输入时自动创建检查点
- 旧检查点最多保留 30 天
- 检查点会自动清理，以防止存储空间无限增长

## 工作流模式

### 探索的分支策略

在探索多种方法时：

```
1. 从初始实现开始 → 检查点 A
2. 尝试方法 1 → 检查点 B
3. 回退到检查点 A
4. 尝试方法 2 → 检查点 C
5. 比较来自 B 和 C 的结果
6. 选择最佳方法并继续
```

### 安全重构模式

在进行重大更改时：

```
1. 当前状态 → 检查点（自动）
2. 开始重构
3. 运行测试
4. 如果测试通过 → 继续工作
5. 如果测试失败 → 回退并尝试不同的方法
```

## 最佳实践

由于检查点是自动创建的，你可以专注于工作而无需担心手动保存状态。不过，请记住以下实践：

### 有效使用检查点

✅ **应该做：**
- 在回退之前查看可用的检查点
- 当你想探索不同方向时使用回退
- 保留检查点以比较不同的方法
- 理解每个回退选项的作用（恢复代码和对话、恢复对话、恢复代码或进行摘要）

❌ **不应该做：**
- 仅依赖检查点来保存代码
- 期望检查点跟踪外部文件系统的更改
- 将检查点用作 git 提交的替代品

## 配置

检查点是 Claude Code 内置的默认行为，不需要任何配置即可启用。每次用户输入都会自动创建一个检查点。

唯一与检查点相关的设置是 `cleanupPeriodDays`，它控制会话和检查点的保留时长：

```json
{
  "cleanupPeriodDays": 30
}
```

- `cleanupPeriodDays`：保留会话历史和检查点的天数（默认：`30`）

> **v2.1.117 更新**：`cleanupPeriodDays` 现在管理四个磁盘缓存的保留，而不仅仅是检查点：
>
> - 会话检查点
> - `~/.claude/tasks/` — 持久化任务列表
> - `~/.claude/shell-snapshots/` — 捕获的 shell 环境快照
> - `~/.claude/backups/` — 滚动设置 / CLAUDE.md 备份
>
> 现在一个设置即可统一在相同天数后清理这四个目录。

## 限制

检查点有以下限制：

- **Bash 命令更改不受跟踪** - 文件系统上的操作如 `rm`、`mv`、`cp` 不会被捕获到检查点中
- **外部更改不受跟踪** - 在 Claude Code 之外（编辑器、终端等）所做的更改不会被捕获
- **不能替代版本控制** - 请使用 git 对代码库进行永久、可审计的更改

## 故障排除

### 检查点丢失

**问题**：找不到预期的检查点

**解决方案**：
- 检查检查点是否已被清理
- 检查磁盘空间
- 确保 `cleanupPeriodDays` 设置得足够高（默认：30 天）

### 回退失败

**问题**：无法回退到检查点

**解决方案**：
- 确保没有未提交的更改冲突
- 检查检查点是否已损坏
- 尝试回退到另一个检查点

## 与 Git 的集成

检查点与 git 互补（但不能替代 git）：

| 特性 | Git | 检查点 |
|---------|-----|-------------|
| 范围 | 文件系统 | 对话 + 文件 |
| 持久性 | 永久 | 基于会话 |
| 粒度 | 提交 | 任意时间点 |
| 速度 | 较慢 | 即时 |
| 共享 | 是 | 有限 |

结合使用两者：
1. 使用检查点进行快速实验
2. 使用 git 提交来保存最终确定的更改
3. 在 git 操作之前创建检查点
4. 将成功的检查点状态提交到 git

## 快速入门指南

### 基本工作流程

1. **正常工作** - Claude Code 自动创建检查点
2. **想要返回？** - 按两次 `Esc` 或使用 `/rewind`
3. **选择检查点** - 从列表中选择以进行回退
4. **选择要恢复的内容** - 从恢复代码和对话、恢复对话、恢复代码、从此处摘要或取消中选择
5. **继续工作** - 你已经回到了那个时间点

### 键盘快捷键

- **`Esc` + `Esc`** - 打开检查点浏览器
- **`/rewind`** - 访问检查点的替代方式
- **`/checkpoint`** - `/rewind` 的别名

## 知道何时回退：上下文监控

检查点让你可以返回——但你如何知道*什么时候*应该返回？随着对话增长，Claude 的上下文窗口被填满，模型质量会悄悄下降。你可能正在用一个半盲的模型来交付代码而不自知。

**[cc-context-stats](https://github.com/luongnv89/cc-context-stats)** 通过在你的 Claude Code 状态栏中添加实时的**上下文区域**来解决这个问题。它跟踪你在上下文窗口中的位置——从**计划区**（绿色，适合规划和编写代码）到**编码区**（黄色，避免开始新计划）再到**转储区**（橙色，完成当前工作并回退）。当你看到区域切换时，你就知道该创建检查点并重新开始，而不是硬撑着用降级的输出继续。

## 总结

检查点是 Claude Code 中的一项自动功能，让你可以安全地探索不同方法而不必担心丢失工作进度。每次用户输入都会自动创建一个新的检查点，因此你可以回退到会话中的任何一个先前位置。

主要优势：
- 无惧地尝试多种方法
- 快速从错误中恢复
- 并排比较不同的解决方案
- 与版本控制系统安全集成

请记住：检查点不能替代 git。请使用检查点进行快速实验，使用 git 进行永久性的代码更改。
