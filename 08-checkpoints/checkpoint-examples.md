# 检查点示例

真实世界中在 Claude Code 中有效使用检查点的示例。

注意：检查点会在每次用户提示时自动创建。你无需手动保存它们。要回退，请按两次 `Esc`（`Esc+Esc`）或使用 `/rewind` 打开检查点浏览器。

## 示例 1：数据库迁移

### 场景
你需要从 MySQL 迁移到 PostgreSQL，但想同时探索直接迁移和渐进式双写两种方法。

### 工作流程

```
用户：让我们实现从 MySQL 到 PostgreSQL 的直接迁移
      [检查点 A 已自动创建]

Claude：我将实现直接迁移...
[更改数据库驱动、更新查询、修改模式]

用户：运行测试
      [检查点 B 已自动创建]

Claude：正在运行测试...
❌ 15 个测试失败 - 连接池问题

用户：这个方法有问题。让我们改用双写模式。
[用户按 Esc+Esc，选择检查点 A，选择"恢复代码和对话"]

用户：现在实现一个双写模式，同时写入两个数据库

Claude：我将实现双写迁移...
[实现双写层、渐进式迁移策略]

用户：运行测试

Claude：正在运行测试...
✅ 所有测试通过

用户：双写方法明显更安全。让我们提交这个。
```

## 示例 8：从检查点使用摘要

### 场景
经过漫长的调试会话后，你想压缩对话同时保留上下文。

### 工作流程

```
用户：[经过 20 多条消息的调试和探索之后]

[用户按 Esc+Esc，选择一个较早的检查点，选择"从此处摘要"]
[可选地提供说明："重点关注我们尝试了什么以及什么有效"]

Claude：[生成从该点开始对话的摘要]
[原始消息保留在记录中]
[摘要替换了可见的对话，减少了上下文窗口的使用]

用户：现在让我们继续用之前有效的方法。
```

## 关键要点

1. **检查点是自动的**：每次用户提示都会创建一个检查点——无需手动保存
2. **使用 Esc+Esc 或 /rewind**：这是访问检查点浏览器的两种方式
3. **选择合适的恢复选项**：根据需要恢复代码、对话、两者或摘要
4. **不要害怕实验**：检查点让大胆的变更变得安全
5. **与 Git 配合使用**：用检查点进行探索，用 Git 进行最终确定的工作
6. **为长会话生成摘要**：使用"从此处摘要"让对话保持可控
