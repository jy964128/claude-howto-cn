# AI 代码生成整洁代码规范

这些规范指导代码生成，以产出可维护、专业质量的代码。

## 有意义的命名
- 使用揭示意图的名称，解释某物为何存在
- 避免误导性信息和无意义的区分（如 `data`、`info`、`manager`）
- 使用可发音、可搜索的名称
- 类名：名词（如 `UserAccount`、`PaymentProcessor`）
- 方法名：动词（如 `calculateTotal`、`sendEmail`）
- 避免心理映射和编码（匈牙利命名法、前缀）

## 函数
- 保持函数小巧（理想情况下 < 20 行）
- 只做一件事——单一职责原则
- 每个函数保持同一抽象层级
- 限制参数数量：0-2 个为佳，最多 3 个，避免布尔标志参数
- 无副作用——函数应如其名称所述行事
- 将命令（改变状态）与查询（返回信息）分离
- 优先使用异常而非错误码

## 注释
- 代码应自解释——尽可能避免注释
- 好的注释：法律信息、警告、TODO、公共 API 文档
- 差的注释：冗余、误导性或为糟糕代码辩解
- 绝不要注释掉代码——删除它（版本控制保留了历史记录）
- 如果需要注释，先考虑重构代码

## 格式规范
- 保持文件小巧且专注
- 垂直格式：相关概念靠近，空行分隔不同概念
- 水平格式：限制行长度（80-120 字符）
- 使用一致的缩进和团队风格
- 将相关函数放在一起

## 对象与数据结构
- 对象：将数据隐藏在抽象之后，通过方法暴露行为
- 数据结构：暴露数据，行为最少化
- 得墨忒耳定律：只与直接的朋友交谈，避免 `a.getB().getC().doSomething()`
- 不要通过 getter/setter 盲目暴露内部结构

## 错误处理
- 使用异常，而非返回码或错误标志
- 当代码可能失败时，先编写 `try-catch-finally`
- 在异常消息中提供上下文
- 不要返回 `null`——返回空集合或使用 Optional/Maybe
- 不要将 `null` 作为参数传递

## 类
- 小类：以职责而非行数衡量
- 单一职责原则：只有一个变更的理由
- 高内聚：类变量被多个方法使用
- 低耦合：类之间的依赖最小化
- 开闭原则：对扩展开放，对修改关闭

## 单元测试
- 快速、独立、可重复、自验证、及时（F.I.R.S.T.）
- 每个测试一个断言（或一个概念）
- 测试代码质量等同于生产代码质量
- 可读的测试名称，描述测试内容
- Arrange-Act-Assert（准备-执行-断言）模式

## 代码质量原则
- **DRY（不要重复自己）**：杜绝重复
- **YAGNI（你不会需要它）**：不要为假设的未来而构建
- **KISS（保持简单）**：避免不必要的复杂性
- **童子军规则**：让代码比你发现时更整洁

## 应避免的代码坏味道
- 过长的函数或类
- 重复代码
- 死代码（未使用的变量、函数、参数）
- 依恋情节（方法对另一个类更感兴趣）
- 过度亲密（类之间了解过多）
- 过长的参数列表
- 基本类型偏执（过度使用基本类型而非小型对象）
- switch/case 语句（考虑多态）
- 临时字段（类变量仅偶尔被使用）

## 并发
- 将并发代码与其他代码分离
- 限制同步/锁定数据的范围
- 使用线程安全的集合
- 保持同步段小而精
- 了解你的执行模型和原语

## 系统设计
- 将构造与使用分离（依赖注入）
- 使用工厂、构建器处理复杂对象创建
- 面向接口编程，而非面向实现
- 优先使用组合而非继承
- 在能简化问题时应用设计模式，而非为了炫耀

## 重构
- 持续重构，而非大批量进行
- 重构前后始终确保测试通过
- 小步前进：每次只改一处
- 常见重构手法：提取方法、重命名、移动、内联

## 文档
- 自文档化的代码 > 注释 > 外部文档
- 公共 API 需要清晰的文档
- 文档中包含示例
- 将文档放在代码附近（理想情况下放在代码中）

---

**核心理念**：代码被阅读的次数是编写的 10 倍。优化可读性和可维护性，而非炫技。
