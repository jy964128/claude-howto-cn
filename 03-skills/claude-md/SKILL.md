```markdown
---
name: claude-md
description: 按照最佳实践创建或更新 CLAUDE.md 文件，实现最优的 AI 代理接入体验
---

## 用户输入

```text
$ARGUMENTS
```

你**必须**在继续之前考虑用户输入（如果非空）。用户可能会指定：
- `create` - 从零创建新的 CLAUDE.md
- `update` - 改进已有的 CLAUDE.md
- `audit` - 分析并报告当前 CLAUDE.md 的质量
- 用于创建/更新的特定路径（例如 `src/api/CLAUDE.md`，用于特定目录的说明）

## 核心原则

**LLM 是无状态的**：CLAUDE.md 是唯一会在每次对话中自动包含的文件。它是 AI 代理接入你代码库的主要入门文档。

### 黄金法则

1. **少即是多**：前沿 LLM 能够遵循约 150-200 条指令。Claude Code 的系统提示已经占用了约 50 条。保持你的 CLAUDE.md 专注且简洁。

2. **普遍适用性**：只包含与**每次**会话都相关的信息。特定任务的说明应放在单独的文件中。

3. **不要用 Claude 做代码检查**：代码风格指南会膨胀上下文并削弱指令遵循能力。应使用确定性工具（prettier、eslint 等）来替代。

4. **永远不要自动生成**：CLAUDE.md 是 AI 工具链中影响力最大的环节。必须手动精心编写。

## 执行流程

### 1. 项目分析

首先，分析当前项目状态：

1. 检查已有的 CLAUDE.md 文件：
   - 根目录级别：`./CLAUDE.md` 或 `.claude/CLAUDE.md`
   - 目录特定：`**/CLAUDE.md`
   - 全局用户配置：`~/.claude/CLAUDE.md`

2. 识别项目结构：
   - 技术栈（编程语言、框架）
   - 项目类型（monorepo、单应用、库）
   - 开发工具（包管理器、构建系统、测试运行器）

3. 审查现有文档：
   - README.md
   - CONTRIBUTING.md
   - package.json、pyproject.toml、Cargo.toml 等

### 2. 内容策略（WHAT、WHY、HOW）

围绕三个维度组织 CLAUDE.md：

#### WHAT - 技术 & 结构
- 技术栈概览
- 项目组织方式（对 monorepo 尤其重要）
- 关键目录及其用途

#### WHY - 目的 & 上下文
- 项目的用途是什么
- 为什么做出某些架构决策
- 每个主要组件的职责是什么

#### HOW - 工作流 & 约定
- 开发工作流（bun vs node、pip vs uv 等）
- 测试流程和命令
- 验证和构建方法
- 关键的"陷阱"或非显而易见的注意事项

### 3. 渐进式披露策略

对于较大的项目，建议创建一个 `agent_docs/` 文件夹：

```
agent_docs/
  |- building_the_project.md
  |- running_tests.md
  |- code_conventions.md
  |- architecture_decisions.md
```

在 CLAUDE.md 中，通过类似以下方式的说明来引用这些文件：
```markdown
有关详细的构建说明，请参阅 `agent_docs/building_the_project.md`
```

**重要提示**：使用 `file:line` 引用而不是代码片段，以避免上下文过时。

### 4. 质量约束

在创建或更新 CLAUDE.md 时：

1. **目标长度**：不超过 300 行（理想情况下不超过 100 行）
2. **不含代码风格规则**：移除任何代码检查/格式化说明
3. **不含特定任务说明**：移至单独的文件
4. **不含代码片段**：使用文件引用代替
5. **不含重复信息**：不要重复 package.json 或 README 中已有的内容

### 5. 必要章节

结构良好的 CLAUDE.md 应包含：

```markdown
# 项目名称

一行简要描述。

## 技术栈
- 主要编程语言及版本
- 关键框架/库
- 数据库/存储（如有）

## 项目结构
[仅适用于 monorepo 或复杂结构]
- `apps/` - 应用程序入口点
- `packages/` - 共享库

## 开发命令
- 安装：`command`
- 测试：`command`
- 构建：`command`

## 关键约定
[仅包含非显而易见的、影响重大的约定]
- 约定 1 及简要说明
- 约定 2 及简要说明

## 已知问题 / 陷阱
[经常让开发者踩坑的事项]
- 问题 1
- 问题 2
```

### 6. 应避免的反模式

**不要包含：**
- 代码风格指南（使用代码检查工具）
- 关于如何使用 Claude 的文档
- 对显而易见模式的冗长解释
- 复制粘贴的代码示例
- 通用的最佳实践（"编写干净的代码"）
- 特定任务的说明
- 自动生成的内容
- 冗长的 TODO 列表

### 7. 验证清单

在最终定稿前，验证：

- [ ] 不超过 300 行（最好不超过 100 行）
- [ ] 每一行都适用于所有会话
- [ ] 没有风格/格式化规则
- [ ] 没有代码片段（使用文件引用）
- [ ] 命令已经过验证可以正常工作
- [ ] 对复杂项目使用了渐进式披露
- [ ] 记录了关键的陷阱
- [ ] 没有与 README.md 重复

## 输出格式

### 对于 `create` 或默认情况：

1. 分析项目
2. 按照上述结构起草 CLAUDE.md
3. 将草稿提交审核
4. 经批准后写入到适当的位置

### 对于 `update`：

1. 阅读已有的 CLAUDE.md
2. 对照最佳实践进行审计
3. 识别：
   - 应移除的内容（风格规则、代码片段、特定任务内容）
   - 应精简的内容
   - 缺少的必要信息
4. 将更改提交审核
5. 经批准后应用更改

### 对于 `audit`：

1. 阅读已有的 CLAUDE.md
2. 生成一份报告，包含：
   - 当前行数与目标行数的对比
   - 普遍适用内容的百分比
   - 发现的反模式清单
   - 改进建议
3. 不要修改文件，仅作报告

## AGENTS.md 处理

如果用户请求创建/更新 AGENTS.md：

AGENTS.md 用于定义专业化的代理行为。与 CLAUDE.md（用于项目上下文）不同，AGENTS.md 定义的是：
- 自定义代理角色和能力
- 特定代理的说明和约束
- 多代理场景的工作流定义

应用类似的原则：
- 保持专注和简洁
- 使用渐进式披露
- 引用外部文档而不是嵌入内容

## 注意事项

- 在纳入命令之前，始终验证其可以正常工作
- 如有疑问，宁可舍弃——少即是多
- 系统提醒告诉 Claude，CLAUDE.md"可能相关也可能不相关"——噪声越多，越容易被忽略
- Monorepo 从清晰的 WHAT/WHY/HOW 结构中获益最大
- 目录特定的 CLAUDE.md 文件应更加聚焦
```
