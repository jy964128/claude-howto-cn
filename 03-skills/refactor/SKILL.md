# 代码重构技能

基于 Martin Fowler 所著 *《重构：改善既有代码的设计》*（第 2 版）的系统化代码重构方法。本技能强调以测试为后盾的安全增量式变更。

> "重构是在不改变代码外部行为的前提下，对其内部结构进行优化的过程。" — Martin Fowler

## 核心原则

1. **行为保持不变**：外部行为必须保持不变
2. **小步前进**：每次只做微小、可测试的变更
3. **测试驱动**：测试是安全网
4. **持续进行**：重构是持续性的，不是一次性事件
5. **协作推进**：每个阶段都需要用户确认

## 工作流概览

```
阶段 1：研究与分析
    ↓
阶段 2：测试覆盖率评估
    ↓
阶段 3：代码异味识别
    ↓
阶段 4：重构计划制定
    ↓
阶段 5：增量实施
    ↓
阶段 6：审查与迭代
```

## 阶段 1：研究与分析

### 目标
- 理解代码库的结构和目的
- 确定重构的范围
- 收集业务需求相关的背景信息

### 需要向用户确认的问题
开始之前，先明确以下内容：

1. **范围**：哪些文件/模块/函数需要重构？
2. **目标**：你想解决什么问题？（可读性、性能、可维护性）
3. **约束条件**：是否有不应该修改的区域？
4. **时间压力**：这是否会阻塞其他工作？
5. **测试状态**：是否存在测试？测试是否通过？

### 操作步骤
- [ ] 阅读并理解目标代码
- [ ] 识别依赖关系和集成点
- [ ] 记录当前架构
- [ ] 标记已存在的技术债标记（TODO、FIXME）

### 输出内容
向用户汇报以下发现：
- 代码结构摘要
- 已识别的问题区域
- 初步建议
- **请求批准继续推进**

## 阶段 2：测试覆盖率评估

### 测试为何重要
> "没有测试的重构，就像不系安全带开车。" — Martin Fowler

测试是安全重构的**关键支撑**。没有测试，你就有可能引入 bug。

### 评估步骤

1. **检查现有测试**
   ```bash
   # 查找测试文件
   find . -name "*test*" -o -name "*spec*" | head -20
   ```

2. **运行现有测试**
   ```bash
   # JavaScript/TypeScript
   npm test

   # Python
   pytest -v

   # Java
   mvn test
   ```

3. **检查覆盖率（如果可用）**
   ```bash
   # JavaScript
   npm run test:coverage

   # Python
   pytest --cov=.
   ```

### 决策点：询问用户

**如果测试存在且通过：**
- 继续进入阶段 3

**如果测试缺失或不完整：**
提供以下选项：
1. 先编写测试（推荐）
2. 在重构过程中逐步添加测试
3. 无测试直接继续（有风险——需要用户明确确认）

**如果测试失败：**
- 停止。在重构之前先修复失败的测试
- 询问用户：我们应该先修复测试吗？

### 测试编写指南（如需要）

对于每个正在重构的函数，确保测试覆盖：
- 正常路径（常规操作）
- 边界情况（空输入、null、边界值）
- 错误场景（无效输入、异常）

使用"红-绿-重构"循环：
1. 编写一个失败的测试（红）
2. 让它通过（绿）
3. 再进行重构

## 阶段 3：代码异味识别

### 什么是代码异味？
代码异味是代码深层问题的表象。它们不是 bug，但提示代码还有改进空间。

### 需要检查的常见代码异味

完整目录请参见 [references/code-smells.md](references/code-smells.md)。

#### 快速参考

| 异味 | 表现 | 影响 |
|-------|-------|--------|
| **过长方法** | 方法超过 30-50 行 | 难以理解、测试和维护 |
| **重复代码** | 相同逻辑出现在多处 | 修复 bug 需要同步改多处 |
| **过大的类** | 类承担了过多职责 | 违反单一职责原则 |
| **依恋情结** | 方法过多地使用另一个类的数据 | 封装性差 |
| **基本类型偏执** | 过度使用基本类型而非对象 | 缺失领域概念 |
| **过长参数列表** | 方法有 4 个及以上参数 | 调用困难，容易出错 |
| **数据泥团** | 相同的数据项总是一起出现 | 缺失抽象 |
| **Switch 语句** | 复杂的 switch/if-else 链 | 难以扩展 |
| **过度泛化** | "以防万一"而写的多余代码 | 不必要的复杂度 |
| **死代码** | 从未使用的代码 | 造成困惑和维护负担 |

### 分析步骤

1. **自动化分析**（如果有脚本可用）
   ```bash
   python scripts/detect-smells.py <file>
   ```

2. **人工审查**
   - 系统地遍历代码
   - 记录每个异味的位置和严重程度
   - 按影响程度分类（严重/高/中/低）

3. **优先级排序**
   重点关注以下异味：
   - 阻碍当前开发的
   - 导致 bug 或困惑的
   - 影响最常修改的代码路径的

### 输出：异味报告

向用户展示：
- 识别的异味清单及其位置
- 每项的严重程度评估
- 建议的优先级顺序
- **请求用户确认优先级**

## 阶段 4：重构计划制定

### 选择重构手法

针对每个异味，从目录中选择合适的重构手法。

完整列表请参见 [references/refactoring-catalog.md](references/refactoring-catalog.md)。

#### 异味与重构手法映射表

| 代码异味 | 推荐重构手法 |
|------------|---------------------------|
| 过长方法 | 提炼函数、以查询取代临时变量 |
| 重复代码 | 提炼函数、方法上移、塑造模板函数 |
| 过大的类 | 提炼类、提炼子类 |
| 依恋情结 | 搬移函数、搬移字段 |
| 基本类型偏执 | 以对象取代基本类型、以类取代类型码 |
| 过长参数列表 | 引入参数对象、保持对象完整 |
| 数据泥团 | 提炼类、引入参数对象 |
| Switch 语句 | 以多态取代条件表达式 |
| 过度泛化 | 折叠继承体系、内联类、移除死代码 |
| 死代码 | 移除死代码 |

### 计划结构

使用 [templates/refactoring-plan.md](templates/refactoring-plan.md) 中的模板。

每项重构应包含：
1. **目标**：将修改哪些代码
2. **异味**：解决什么问题
3. **重构手法**：应用哪种技术
4. **步骤**：详细的微步骤
5. **风险**：可能出现什么问题
6. **回滚**：必要时如何撤销

### 分阶段方法

**关键提示**：重构需要分阶段逐步推进。

**阶段 A：快速见效**（低风险，高价值）
- 重命名变量以增强清晰度
- 提取明显重复的代码
- 移除死代码

**阶段 B：结构改进**（中等风险）
- 从过长函数中提取方法
- 引入参数对象
- 将方法移到更合适的类中

**阶段 C：架构变更**（较高风险）
- 以多态取代条件表达式
- 提炼类
- 引入设计模式

### 决策点：向用户展示计划

实施之前：
- 展示完整的重构计划
- 解释每个阶段及其风险
- 获取每个阶段的明确批准
- **询问**："我可以开始执行阶段 A 吗？"

## 阶段 5：增量实施

### 黄金法则
> "修改 → 测试 → 变绿？→ 提交 → 下一步"

### 实施节奏

对每个重构步骤：

1. **预检查**
   - 测试通过（绿色）
   - 代码可以编译

2. **只做一个**小改动
   - 按照目录中的操作步骤来
   - 保持改动最小化

3. **验证**
   - 立即运行测试
   - 检查编译错误

4. **如果测试通过（绿色）**
   - 使用描述性信息提交
   - 继续下一步

5. **如果测试失败（红色）**
   - 立即停止
   - 撤销改动
   - 分析出错原因
   - 不清楚时询问用户

### 提交策略

每次提交应当：
- **原子性**：一个逻辑变更
- **可回滚**：易于撤销
- **描述清晰**：提交信息明确

示例提交信息：
```
refactor: 从 processOrder() 中提取 calculateTotal()
refactor: 将 'x' 重命名为 'customerCount' 以增强清晰度
refactor: 移除未使用的 validateOldFormat() 方法
```

### 进度汇报

每个子阶段完成后，向用户汇报：
- 所做的变更
- 测试是否仍然通过？
- 遇到的任何问题
- **询问**："继续下一批吗？"

## 阶段 6：审查与迭代

### 重构后检查清单

- [ ] 所有测试通过
- [ ] 没有新的警告/错误
- [ ] 代码编译成功
- [ ] 行为未发生变化（手动验证）
- [ ] 文档已按需更新
- [ ] 提交历史清晰

### 指标对比

在重构前后运行复杂度分析：
```bash
python scripts/analyze-complexity.py <file>
```

展示改进效果：
- 代码行数变化
- 圈复杂度变化
- 可维护性指数变化

### 用户审查

展示最终结果：
- 所有变更摘要
- 重构前后代码对比
- 指标改进情况
- 剩余的技术债
- **询问**："你对这些变更满意吗？"

### 后续步骤

与用户讨论：
- 还有需要处理的异味吗？
- 是否安排后续的重构？
- 是否在其他地方应用类似的变更？

## 重要指南

### 何时暂停并询问

遇到以下情况时务必暂停并与用户商议：
- 不确定业务逻辑
- 变更可能影响外部 API
- 测试覆盖率不足
- 需要做出重大的架构决策
- 风险等级上升
- 遇到意料之外的复杂性

### 安全规则

1. **没有测试就不要重构**（除非用户明确确认风险）
2. **不要做大改动**——拆分成微小的步骤
3. **每次改动后不要跳过测试**
4. **测试失败就不要继续**——先修复或回滚
5. **不要猜测**——有疑问就问

### 不该做的事

- 不要把重构与新功能开发混在一起
- 不要在生产事故期间进行重构
- 不要重构你不理解的代码
- 不要过度设计——保持简洁
- 不要一次性重构所有内容

## 快速入门示例

### 场景：过长方法 + 代码重复

**重构前：**
```javascript
function processOrder(order) {
  // 150 行代码，包含：
  // - 重复的验证逻辑
  // - 内联计算
  // - 职责混杂
}
```

**重构步骤：**

1. **确保测试存在**，覆盖 processOrder()
2. **将验证逻辑**提取为 validateOrder()
3. **测试**——应当通过
4. **将计算逻辑**提取为 calculateOrderTotal()
5. **测试**——应当通过
6. **将通知逻辑**提取为 notifyCustomer()
7. **测试**——应当通过
8. **审查**——processOrder() 现在编排了 3 个清晰的函数

**重构后：**
```javascript
function processOrder(order) {
  validateOrder(order);
  const total = calculateOrderTotal(order);
  notifyCustomer(order, total);
  return { order, total };
}
```
